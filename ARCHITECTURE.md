# Citeopolis — Architecture Overview

## What this project is

Citeopolis is a **React frontend for WordPress multisite**. A single codebase serves multiple municipal websites — each with a different domain, visual theme, and set of active features (modules like News, Albums, Flash Info). The backend (WordPress + WPGraphQL) determines which theme and features each site gets. The frontend adapts at runtime.

---

## Monorepo structure

```
citeopolis-migration/
├── packages/               # Reusable feature packages (npm scope: @packages/*)
│   ├── news/               # @packages/news
│   ├── albums/             # @packages/albums
│   └── flash-info/         # @packages/flash-info
├── template/               # The React application
│   └── src/
│       ├── lib/            # Core systems (autoloader, themes, routing utilities)
│       ├── modules/        # Template-side module extensions (pages only)
│       ├── themes/         # Theme variants (base, municipal, metropolitain…)
│       ├── components/     # Default UI components (layout, blocks)
│       ├── routes/         # TanStack Router file-based routes
│       ├── pages/          # Page components (one per GraphQL route typename)
│       └── mock/           # Apollo mock link (no real backend needed)
└── sites/                  # Per-client site overrides (optional)
```

The monorepo uses **pnpm workspaces**. The `template` application declares feature packages as regular npm dependencies using the `workspace:*` protocol — exactly like any npm package, but resolved locally.

## The three independent systems

The architecture has three orthogonal systems. They can be reasoned about separately.

### 1. Feature packages — what blocks exist

Feature packages (`@packages/news`, `@packages/albums`, `@packages/flash-info`) are npm-style packages that live under `packages/`. Each one ships:

- **React components** — the blocks and UI for that feature
- **A manifest** (`src/manifest.ts`) — declares which GraphQL `__typename` maps to which block component

```
packages/news/src/
├── index.ts          ← public API (components, types)
├── manifest.ts       ← { slug: "news", blocks: { NewsBlock: () => import(…) } }
├── blocks/           ← block components
└── types.ts
```

**Installing / uninstalling a package directly gates its blocks.**

```bash
pnpm add @packages/news@workspace:*    # → NewsBlock becomes available
pnpm remove @packages/news             # → NewsBlock disappears (after restart)
```

This works via a **Vite virtual module** (`virtual:@packages/manifests`) generated by `vite-plugin-modules.ts`. At dev server start, the plugin reads `template/package.json`, finds every `@packages/*` dependency that exports a `./manifest` entry, and generates:

```ts
// virtual:@packages/manifests  (auto-generated at build time)
import m0 from "@packages/news/manifest";
import m1 from "@packages/albums/manifest";
export default [m0, m1];
```

The autoloader (`src/lib/autoloader.ts`) imports this virtual module and registers all block components into the **block registry** — a `Map<GraphQL __typename, React component>`.

Local manifests in `src/modules/*/manifest.ts` are only for **page components** (template-specific concerns), not blocks.

---

### 2. Site factory — which features are active per domain

Each domain maps to a site configuration. This is the **multi-tenancy layer**.

`src/lib/siteConfigs.ts` defines a config per hostname:

```ts
{ hostname: "localhost",           theme: "municipal", features: ["news", "albums", "flash-info"] }
{ hostname: "metropolis.localhost", theme: "metropolis", features: ["albums"] }
{ hostname: "base.localhost",       theme: "base",       features: [] }
```

On startup, `App.tsx` queries the backend (`GetSiteConfig`) for the active site's `theme` and `features`. The local `siteConfigs.ts` acts as a fallback for local development when the backend is mocked or not yet updated.

The `features` array flows into React context (`FeaturesContext`) and is consumed by two places:

1. **`BlockRenderer`** — skips rendering any block whose feature slug is not in the active `features` list
2. **`FlashInfoBar`** — returns null immediately if `"flash-info"` is not in `features`

This means: even if a block component is installed (registered in the block registry), it will not render on a site that has not declared the feature. **Installing a package makes the block available; the site's feature list controls whether it actually appears.**

```
Installed packages  →  blockRegistry (all possible blocks)
                              ↓
              features array filters at render time
                              ↓
                   Only active blocks render
```

### 3. Theme system — what components look like

Each site has a theme. Themes can **inherit** from each other (e.g. `municipal` extends `base`).

#### CSS tokens

The active theme is applied as a `data-theme` attribute on `<html>`. Each theme defines CSS custom properties (colors, fonts, spacing) in SCSS. Changing `data-theme` hot-swaps the visual identity.

#### Component overrides

Themes can override any shared UI component. The resolution order is:

```
sites/{site}/overrides/components/   ← highest priority (client-specific)
themes/{theme}/components/           ← theme override
themes/{parent}/components/          ← parent theme (walks chain)
src/components/                      ← base fallback
```

This is resolved at **build time** by the `themePlugin` Vite plugin, which intercepts `@components/*` imports and rewrites them to the correct file in the chain. At runtime, `componentRegistry.ts` provides `resolveComponent()` for dynamic lazy-loading.

**Example:** The `Header` component in `municipal` theme overrides the default header. Any site using `municipal` theme automatically gets it. A child theme (e.g. `municipal-dark`) would get it too unless it defines its own header.

## Data flow on page load

```
Browser → domain: metropolis.localhost
                │
                ▼
         App.tsx starts
                │
                ├─ 1. Queries GetSiteConfig
                │       → theme: "metropolis"
                │       → features: ["albums"]
                │
                ├─ 2. Applies theme to <html data-theme="metropolis">
                │
                ├─ 3. Provides FeaturesContext = ["albums"]
                │
                └─ 4. RouterProvider renders RootLayout
                            │
                            ├─ Header resolved via theme chain
                            ├─ FlashInfoBar → "flash-info" not in features → null
                            │
                            └─ Page content (Outlet)
                                        │
                                        ▼
                               TanStack Router catches URL
                                        │
                                        ▼
                               $.tsx loader: GET_ROUTE query
                               → typename: "Page"
                                        │
                                        ▼
                               PagePage renders
                               → GET_PAGE query
                               → structuredContent: [AlbumsBlock]
                                        │
                                        ▼
                               BlockRenderer maps typename → component
                               → "AlbumsBlock" → in features? YES → renders
                               → "NewsBlock"   → not registered (package not installed)
```

## The routing model

The app uses **TanStack Router** with a single catch-all route (`$`).

Every URL hits `$.tsx`, which queries the backend `route(url)` resolver. The backend returns a GraphQL `__typename` that tells the frontend what kind of content lives at that URL:

| `__typename` returned | Component rendered |
|---|---|
| `Page` | `PagePage` (generic WordPress page with `structuredContent`) |
| `News` | `NewsPage` (news detail view) |
| `Redirect` | Browser redirect (no component) |

Page components for feature types (`News`, etc.) are registered in the **page registry** by local manifests in `src/modules/*/manifest.ts`. The root route (`/`) is handled separately by `routes/index.tsx`.

---

## Blocks — how a Gutenberg block becomes a React component

WordPress content is stored as structured blocks. The GraphQL API returns `structuredContent` as an array where each item has a `__typename` (e.g. `NewsBlock`, `AlbumsBlock`).

`BlockRenderer` maps each `__typename` to a React component via the **block registry**, applies the feature gate, and renders with `<Suspense>` for lazy loading:

```
structuredContent: [
  { __typename: "NewsBlock", listUrl: "/actualites", ... }
  { __typename: "AlbumsBlock", listUrl: "/albums", ... }
]
         │
         ▼
BlockRenderer iterates
  "NewsBlock"   → features.includes("news")?  YES → blockRegistry.get("NewsBlock")  → <NewsBlock />
  "AlbumsBlock" → features.includes("albums")? YES → blockRegistry.get("AlbumsBlock") → <AlbumsBlock />
```

## Mock API

During local development (when `VITE_MOCK_API=true`), a custom **Apollo Link** intercepts all GraphQL queries and returns demo data. No real backend is needed.

The mock (`src/mock/handlers.ts`) reads the site config for the current hostname and builds content accordingly:
- `GetSiteConfig` → returns `theme` and `features` from `siteConfigs.ts`
- `GetPage` → builds homepage blocks from the site's active features
- `GetRoute` → returns `Page` typename for all URLs
- `GetFlashInfos` → returns flash-info items only if `"flash-info"` is in features

This allows testing all multi-site scenarios locally without deploying the backend.

## Adding a new feature — complete checklist

1. **Create the package** under `packages/{name}/` with `src/index.ts`, `src/manifest.ts`, and `package.json` (named `@packages/{name}`)
2. **Add a manifest** in the package (`src/manifest.ts`) declaring block component loaders keyed by GraphQL `__typename`
3. **Install it** in the template: `pnpm add @packages/{name}@workspace:*` (from `template/`)
4. **Restart the dev server** — the virtual module picks it up automatically
5. **Declare the feature** in `siteConfigs.ts` for whichever sites should have it
6. *(Optional)* Create `template/src/modules/{name}/manifest.ts` **only** if the feature needs a page component (detail view reached via GraphQL `route()`)

To remove a feature from a site: remove it from `features` in `siteConfigs.ts`. To remove it entirely: `pnpm remove @packages/{name}` + delete the package directory.
